Core principles of Indiebox

Indiebox was developed according to the following principles:

  • Content data remains on the system.
  • There is no hidden transfer of content or usage data to the vendor.
  • The system can be operated fully offline and air gapped.
  • Control over models, capabilities, and integrations remains with the owner or the system administrator.

“Content” means your own data such as prompts, chat histories, uploads, documents, generated outputs, and derived indexes. These remain physically on the box. Compliance can be verified through architecture, deployed components, and control of outbound network traffic.

The feature scope available to users is defined by administrative settings. Changes to models or system-wide capabilities are made only by authorized administrators.

How Indiebox works

Prompts, documents, and responses are processed locally. There is no hidden content telemetry and no forwarding of content or usage data.

“Hidden telemetry” means mechanisms where systems send information such as prompts, inputs, usage patterns, or system status to external servers in the background without functional necessity or transparent configuration. This does not happen here.

Technical network traffic can still exist, for example for infrastructure protocols like DNS or NTP, or to make the system reachable. Outbound internet traffic occurs only when:

  • updates are triggered
  • external information sources are actively used
  • integrations or tools are deliberately configured and used

There is no autonomous transfer of content or usage data to the internet.

Offline and air-gapped operation

The system is preinstalled and ready to use. No additional installation is required for core functions. For commissioning, the system is either connected to an existing network or provides its own local Wi-Fi. Fully isolated operation without internet access is also possible.

Chat, analysis of provided data, and content generation work without external dependencies. Models are installed locally and ready to run.

Updates can also be performed in air-gapped setups by transferring and applying verified update packages in a controlled way.

Models, control, and censorship

Indiebox uses established open models that run locally. There is no downstream filtering, logging, or content control by the vendor.

System behavior regarding response style, content restraint, or limitations is determined by the selected model itself. There is no central instance monitoring or modifying responses.

Which models are installed, removed, or used in production is defined by administrators. Regular users cannot change this configuration on their own.

Models can be exchanged. The assessment of quality, suitability, and potential content behavior remains the responsibility of the owner or administrator.

Integrations and external access

By default, content data remains local. External integrations are optional and must be actively configured, for example for:

  • web research
  • external AI services
  • databases, APIs, or MCP
  • internal company systems and network drives

Before enabling such functions, assess the impact on data protection, information leakage, and system behavior. During web research, information about the searched topic inevitably becomes visible externally. When external services are connected, data leaves the local system.

Prompt injection risk must also be considered. In simple chat scenarios this is generally manageable. For agents with broader system or data access, insufficient validation of external content can lead to manipulation. Using these features therefore requires deliberate risk assessment.

Multi-user operation and roles

Indiebox supports multi-user operation. User areas are separated from each other. Content is not visible to other users unless corresponding permissions are granted.

Administrative accounts have system-level access rights, as with any locally operated IT system. This control remains fully within your own organization.

Roles, permissions, and model access are defined and managed locally.

Data storage, retention, and deletion

Indiebox stores working data locally, for example chats, uploads, and generated indexes, depending on usage.

Data storage is technically traceable and takes place exclusively within your own infrastructure. Deletion and retention rules are controlled at system or object level, for example by removing chats, projects, or datasets. Selective deletion of individual items within an existing chat history is not always supported by design.

There is no parallel or hidden storage outside the system.

GDPR in practice

In purely local operation, no new external processors are added. There is no automatic integration of external AI providers or cloud services. This means no additional data processing agreement arises solely from using the system.

Processing takes place within your own infrastructure, comparable to a self-operated database, file server, or document management system. Responsibilities remain in your organization.

If personal data is processed, this is a processing activity under GDPR. Legal assessment, purpose definition, and retention periods remain the responsibility of the data controller. As long as no external integrations are activated, the recipient scope is technically limited to your own organization.

FAQ

Can I verify what is running on my system?

Yes. You can verify at any time which models are installed and which software components are running. Model versions in this setup are referenceable through unique digests/hashes: if an expected digest is present, you can verify that exactly this model version exists locally and is being used. The same applies to stack components: they run in your network and can be monitored by your team.

How do I know as a private user that my behavior is not analyzed?

Indiebox is designed for no behavioral analysis and no telemetry about usage patterns. If maximum assurance is required, run the system offline or block outbound connections at router/firewall level. In that setup, the system cannot transmit usage data even if there were a defect.

Do my contents ever leave the system?

Not by default. Content leaves the system only when external features are enabled, such as external AI services, tools, or connectors. An admin decides whether such connections exist and which data sources they use.

What happens if I enable tools, search, MCP services, or connectors?

Then Indiebox can deliberately interact with your environment or external services, which is exactly what these integrations are for. Connections work only when configured (targets, credentials, permissions). This determines which systems are reachable and which data may appear in workflows.

Where is my data stored, and how can I remove it?

Data is stored locally on the box, including derived data like indexes. Deletion is therefore a local operations topic: you control what is stored, how long it remains, and when it is removed.
